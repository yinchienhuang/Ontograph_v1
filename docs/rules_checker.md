# LLM Rule Checker — Usage Guide

The rule checker detects constraint violations in the constructed ontology using LLMs
instead of an OWL reasoner.  It supports two modes designed for research comparison:

| Mode | Input | What the LLM sees |
|------|-------|-------------------|
| **ontology** | Working OWL file | Exact property values extracted from the OWL graph |
| **document** | Synthesized markdown | Deliberately vague plain-English rule description |
| **both** | OWL + document | Both — side-by-side results for comparison |

The research question: *does the LLM do better when it has precise structured data (ontology)
vs. vague documentation prose (document)?*

---

## Quick start

```bash
# 1. Ontology mode — check rules against the working OWL graph
.venv/Scripts/python scripts/check_rules.py \
    --rules    data/rules_example.yaml \
    --owl      data/ontology/working.owl \
    --provider claude \
    --mode     ontology

# 2. Document mode — check using synthesized prose (no OWL needed)
.venv/Scripts/python scripts/check_rules.py \
    --rules    data/rules_example.yaml \
    --doc      data/raw/cubesatontology_synthesized.md \
    --provider claude \
    --mode     document

# 3. Both modes side-by-side
.venv/Scripts/python scripts/check_rules.py \
    --rules    data/rules_example.yaml \
    --owl      data/ontology/working.owl \
    --doc      data/raw/cubesatontology_synthesized.md \
    --provider claude \
    --mode     both
```

Results are printed as a table and saved to `data/violations/<id>_violations.json`.

## YAML rules file format

```yaml
namespace: "http://example.org/cubesat-ontology#"   # OWL namespace prefix

rules:
  - id: "compat-001"
    name: "Battery Cold-Case Temperature Rating"
    subject_type: "PowerSubsystem"   # OWL class local name — can be null
    object_type:  "Battery"          # OWL class local name — null = single-entity rule
    when:
      attribute: "operatingTempMinC" # datatype property local name
      operator:  ">"                 # >, <, >=, <=, ==, !=
      value:     -20.0               # threshold value
      unit:      "°C"                # optional unit label (shown in LLM prompt)
    consequence: "notCompatibleWith" # what violation means (for reporting)
    severity:    "critical"          # critical | warning | info
    note: >
      Human-readable explanation stored verbatim and passed to the LLM.
    plain_english: ""                # leave blank — auto-generated by the LLM
```

### Two rule shapes

**Pair rule** (`object_type` set): checks each (subject, object) combination.
The `when` condition is evaluated on the **object** individual.

```
subject_type: "PowerSubsystem"   → EPS1
object_type:  "Battery"          → BatteryPack1
Checked: BatteryPack1.operatingTempMinC > -20  → VIOLATED (-10 > -20)
```

**Single-entity rule** (`object_type: null`): checks each subject individual on its own.
The `when` condition is evaluated on the **subject** individual.

```
subject_type: "OnBoardComputer"  → OBC1
object_type:  null
Checked: OBC1.powerW > 1.5  → VIOLATED (2.0 > 1.5)
```

## Running inside the full pipeline

Add `--rules` (and optionally `--rules-mode`) to any `run_pipeline.py` invocation.
The rule check runs as **Step 7**, after the OWL is written:

```bash
.venv/Scripts/python scripts/run_pipeline.py \
    --from-owl data/ontology/cubesatontology.owl \
    --owl      data/ontology/working.owl \
    --provider claude \
    --auto-approve \
    --rules      data/rules_example.yaml \
    --rules-mode both
```

## plain_english and the document mode

Before a document-mode check, the LLM generates a `plain_english` description for each
rule that has an empty `plain_english` field.  This description is **deliberately vague**
— it omits exact thresholds and property names — simulating what an engineer might write
in a design document without complete specifications.

**Example:**
- Structured rule: `Battery.operatingTempMinC > -20 °C → notCompatibleWith PowerSubsystem`
- Generated plain English: *"Batteries used in this mission should be rated for the cold
  thermal environment encountered during eclipse.  Engineers should verify that selected
  battery units are suitable for the expected temperature range."*

The research comparison runs the LLM against both versions and records whether the correct
violation is detected in each case.

## Output format

Results are saved to `data/violations/<report_id>_violations.json`:

```json
{
  "id": "a1b2c3d4e5f6a7b8",
  "created_at": "2026-02-25T12:00:00+00:00",
  "mode": "ontology",
  "violations": [
    {
      "rule_id": "compat-001",
      "severity": "critical",
      "mode": "ontology",
      "subject_label": "EPS1",
      "object_label": "BatteryPack1",
      "violated": true,
      "confidence": 0.97,
      "explanation": "BatteryPack1.operatingTempMinC = -10°C, which is above the -20°C threshold. The battery cannot survive LEO eclipse temperatures."
    }
  ]
}
```

Use `ViolationReport.critical()` to filter for `violated=True` entries, and
`ViolationReport.by_rule("compat-001")` to inspect results for a specific rule.
